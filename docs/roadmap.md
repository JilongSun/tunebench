# 路线图

本文档描述 TuneBench 在当前阶段之后的推进方向。当前已经完成一轮面向双后端分类任务的结构收敛，因此路线图不再承担“临时重构计划”职责，而是只描述下一阶段要做什么。

## 当前基线

项目当前已经具备以下基线：

- BERT 路径完整闭环已经稳定。
- LlamaFactory 路径已经完成训练、导出、独立评测与 chat 闭环。
- CLI 参数语义、训练 run metadata、资产路径和 reasoning 语义已经完成第一轮统一。
- LlamaFactory 后端已经按模型能力、推理加载、生成执行、训练恢复和生命周期写回完成拆分。

这意味着接下来的工作重点不再是“继续写重构计划”，而是基于已经稳定下来的边界继续扩展能力并硬化质量。

## 主线一：扩展模型覆盖与运行能力

### 模型覆盖目标

- 扩展更多可验证的 Qwen 模型映射与模板支持。
- 继续提升 train、evaluate、chat 在多后端之间的一致性。
- 在不破坏现有资产与 metadata 约束的前提下增加新能力。

### 模型覆盖重点方向

- 补齐更多模型键的官方可验证映射。
- 明确不同模型族的 loader family、processor 装载与 prompt 模板语义。
- 继续完善 external chat 与 internal run chat 的能力矩阵。

## 主线二：收紧 reasoning 与评测语义

### 语义一致性目标

- 保证 train、evaluate、chat 对 reasoning_mode、template 和 suffix 控制的理解保持一致。
- 让结构化评测结果在多模型、多模板下仍保持可比较性。

### 语义一致性重点方向

- 继续收敛 Qwen3 与 Qwen3.5 的 no_think 训练、评测和 chat 语义。
- 继续清晰区分 native `enable_thinking` 与模板链的 reasoning 控制。
- 继续完善原始输出、清洗输出和生成统计字段的使用边界。

## 主线三：稳固质量与可维护性

### 维护性目标

- 在现有边界上持续降低维护成本，而不是重新打开大规模结构重构。
- 让新增功能优先复用既有模块，而不是回流到兼容层或超大文件。

### 维护性重点方向

- 继续保持 `cli.py` 的接口层定位，新增复杂参数语义优先落到 `cli_support.py` 或对应后端 helper。
- 继续保持 `run_metadata.py`、路径管理和 lifecycle helper 的单一职责。
- 避免在 `inference.py`、`models.py` 这类兼容层文件中新增业务逻辑。
- 通过文档、静态检查和后续测试补齐维护基线。

## 抽象策略

下一阶段仍然遵循以下原则：

- 先让真实需求推动抽象，而不是为未知后端提前建设复杂平台。
- 只有同时改善 BERT 与 LlamaFactory 两条路径的抽象，才值得保留。
- 当现有模块边界足够承载需求时，优先在原边界内扩展，而不是重新洗牌目录结构。

不建议做的事：

- 为未来未知模型提前建设复杂插件系统。
- 为了形式上的“平台化”再次打散已经稳定下来的模块。
- 在没有新增需求的前提下重开一轮大规模结构重构。

## 下一阶段完成标准

近期可以按以下标准判断阶段性完成：

1. 更多模型映射经过验证并稳定开放。
2. reasoning 相关参数在 train、evaluate、chat 三条链路上保持一致语义。
3. 新增功能继续复用现有 CLI、metadata、路径与 lifecycle 边界，没有明显回流到兼容层。
4. 正式文档能够持续准确反映项目现状与下一阶段目标。
